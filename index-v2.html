PSV-IM Agent

Autonomous incident manager for Capillary's Professional Services org.

10-stage async pipeline · code-grounded RCA · GitHub App auth · Facets-ready

What it does

Watches PSV tickets, classifies the concern, pulls the brand's source repo, posts a code-grounded RCA + suggested fix, and tracks SLA.

How tickets enter the pipeline

Webhook (instant)

JIRA POSTs to /webhook/jira on issue create / update / comment. HMAC-validated against WEBHOOK_SECRET. ~10s to first agent comment.

POST /webhook/jira
X-Hub-Signature: sha256=…
{ "webhookEvent": "jira:issue_created",
  "issue": { "key": "PSV-30534", … } }

Poller (3-min fallback)

Polls JIRA every 180s with JQL filtering. Catches anything the webhook missed. Shared dedup set with the webhook so the same ticket can't be processed twice.

JQL: project = PSV
     AND priority in ("Highest-P0","High-P1")
     AND issuetype in (Bug, "Change Request", "P0 Alert")

Correction re-run

When a human comments on a ticket the agent already analysed, the agent re-runs Stage 3a + 5 with the human correction folded into the prompt. Only fires for non-agent comments to avoid loops.

Resume (state-aware)

Workflows persist as JSON on PVC. Agent restart → in-flight tickets resume at their last completed stage. Approval-gate breaches auto-escalate on next monitor tick.

Stage 3d — Code grounding (the differentiator)

Without grounding, RCAs reason in the abstract and hallucinate file paths. With grounding, every hypothesis cites real source.

The flow

1. Parse extracts ticket tokens (file refs, API paths, lambda names)
2. Look up brand_repos for the ticket's Brand field
3. Shallow-clone the repo (depth=200, ~2-3min cold; ~2s warm fetch)
4. git grep --fixed-strings for each token
5. git log --since="30 days ago" for recent commits touching matched paths
6. Assemble code_grounding state → feeds into 5-Whys prompt

Auth in production: GitHub App → JWT → ~1h installation token. Token never persists to .git/config.

Security — 3 push-prevention layers

Defense in depth. Even if Layer 1 is misconfigured, Layers 2 and 3 stop a write.

#LayerMechanismIf it fails
1 External GitHub App scoped Contents: Read. Token can't push by definition. Layers 2 + 3 still block.
2 Config After every clone/fetch, git remote set-url --push origin do-not-push://psv-im-agent-is-read-only/ Layer 3 still blocks.
3 Code Every git subcommand checked against _ALLOWED_GIT_SUBCOMMANDS allowlist. push, commit, merge, rebase, tag, submodule raise before exec. —

Credential hygiene

  • ✓ Authed clone URL never persists to .git/config — wiped immediately after clone
  • ✓ Installation token lives in process memory only (~1h TTL)
  • ✓ Secret + PII redaction on every grep match (SECRET_PATTERNS: AWS keys, GH tokens, JWTs, PEMs, emails, phones)
  • ✓ /admin/test-pull verifies credential_in_config: false per brand
Verified end-to-end: GitHub clone of Capillary/abbott-global finished in 2m20s. Push URL = do-not-push://…. Zero credentials in .git/config.

Quality gates & SLA

Coverage gate (between Stage 2b and Stage 4)

If classification confidence is low AND no BRD/SDD coverage was found, the agent refuses to post speculative findings. It escalates with psv-im-needs-human instead.

if classification.confidence < 0.30 AND
   context.brd_coverage == NOT_COVERED AND
   context.sdd_coverage == NOT_COVERED:
       → escalate ("Needs Human Investigation")

Approval gate (Stage 7)

PrioritySLARemindersBreach action
P0 (Highest)24 hours50% / 75% / 95%auto-escalate to psv-im-needs-human
P1 (High)72 hours50% / 75% / 95%auto-escalate

Reminder severities escalate: warning → warning → critical. Idempotent — restart can't re-fire prior nudges.

QA verdict (Stage 6)

Template-based scoring (out of 13 criteria). Score <9/13 → retry up to 2x. Score <9/13 after 2 retries → escalate.

Guardrails & safety nets

ConcernGuardrail
Speculative findings Coverage Gate — escalates to human when confidence is low. Never posts a "best guess" RCA.
Bad fix applied Approval gate (Stage 7) — fix lands in JIRA as a comment, awaits human approved / rejected reply. SLA-bounded with auto-escalation.
Output leaks services/guardrails.py — scrubs unverified specifics, redacts secrets/PII before publishing comments.
Code-side credential leak 3-layer push prevention (App scope · push URL rewrite · subcommand allowlist).
Anthropic outage Sonnet → Haiku fallback. If both fail, agent posts Coverage Gate escalation instead of stub findings.
Wrong RCA after correction Correction-driven re-run. Human comment triggers Stage 3a + 5 re-execution; agent retracts prior comment explicitly with a new "Correction" comment.
Stuck workflows Approval-gate auto-escalation on SLA breach. State persisted to PVC so restarts don't lose progress.
Compromised installation token Tokens are short-lived (~1h). Process-memory only. Rotation = generate new App key, restart pod (~15min, no downtime for in-flight workflows).
