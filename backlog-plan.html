Phase A — Backlog sweep mode

~1 day of work · agent-side only · ships in one commit

A1

JQL sweep query in agents/jira_poller.py

One paginated JQL pulls every open PSV ticket without the psv-im-processed label, queues them in oldest-first order, processes through the existing pipeline.

~60 LOC · uses existing JiraService.search
A2

CLI flag --sweep-backlog in main.py

One-shot manual trigger so the first sweep can be run, observed, and aborted if anything looks off — before automating.

~25 LOC
A3

Four new env vars in config.py

BACKLOG_SWEEP_ENABLED · BACKLOG_SWEEP_PRIORITIES · BACKLOG_SWEEP_HOUR · BACKLOG_SWEEP_LABEL. Documented in .env.example.

~10 LOC + docs

Dedup mechanism — already exists

Stage 0 of the pipeline already applies psv-im-processing → psv-im-resolved. Adds a final psv-im-processed label. JQL excludes this label → ticket is never re-processed unless the label is removed (e.g. by a human asking for re-analysis).

The sweep JQL

project = PSV
AND issuetype in ("Bug", "Change Request", "P0 Alert")
AND status in ("Open", "To Do", "Reopened", "Under review")
AND priority in (Highest-P0, High-P1, Medium-P2)
AND (labels is EMPTY OR labels not in (psv-im-processed))
ORDER BY priority DESC, created ASC

Design choices baked into this JQL

  • Closed tickets excluded — only Open/To Do/Reopened/Under review. No zombie comments on resolved work.
  • Oldest-first within priority — fairer; clears long-pending tickets sooner.
  • Label-based dedup — once psv-im-processed is on a ticket, it's invisible to the sweep until a human removes it.
  • P2 included by default — manager said "Incident Bugs" without a priority qualifier; can be tightened via env var if cost is a concern.

Phase B — Nightly sweep cron

~half a day · catches anything the live webhook missed

B1

Background task in main.py

Daemon checks every 60s; when hour matches BACKLOG_SWEEP_HOUR, kicks off one sweep cycle. Default: 02:00 IST (low-traffic, mid-day for US ops).

~30 LOC · runs alongside SLA monitor + snapshot scheduler

Why nightly catches matter

  • Webhook misses happen — Jira can drop deliveries under load
  • Agent restarts mid-burst may queue-drop tickets in flight
  • Late-arrival tickets created during a backlog sweep get caught the next day
  • Self-healing — operator doesn't need to manually re-sweep after every incident

Three risks worth flagging

💰 Claude cost spike

Sweeping a 200-ticket backlog · ~2 Claude calls per ticket via the pipeline.

Mitigation: Use Haiku for triage/classify (already), Sonnet only for RCA. Estimate spend before each sweep.

🚦 Jira rate limits

Atlassian rate-limits at ~10 req/s per token. 5 concurrent × ~6 Jira calls each = 1200+ calls per sweep.

Mitigation: asyncio.Semaphore on JiraService capped at 4 req/s.

⚠ Wrong AI comments on old tickets

Once posted, comment is visible to the reporter. Stale tickets may have outdated context.

Mitigation: "AI-generated, please review" prefix on backlog comments specifically; lower posting threshold for stale tickets.
