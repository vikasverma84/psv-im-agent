Step 1-2 — Setup the context

STEP 1

Brand selection

Searchable dropdown across all 246 brands. The choice routes credentials, SDDs, ticket fields, and AI prompts.

⚠ If your brand is missing, ask a lead to add it — don't pick a similar one.

STEP 1a

Brand credentials

Cluster + till_id + password (MD5-hashed, never re-displayed). Saved per user, per brand — first person to add a brand sets it once for the team.

STEP 1b

Existing or new?

Existing ticket → enter the key (e.g. PSV-30533). Chatbot fetches details + runs the auto-process pipeline.

STEP 1c · NEO ONLY

Inline Neo API tester

Pick HTTP method, paste endpoint path (base URL {cluster}/x/neo/ auto-prefixed), paste body, auth auto-applied. Faster than Postman.

STEP 2

SDD / BRD collection

Three branches: paste Confluence links · upload PDF/TXT/DOC · let chatbot search Confluence (SA + AG spaces).
All saved per-brand and shared with the whole team.

Why this matters: a concern classified without SDD context defaults to confidence 0.3 and produces generic AI output. 2 minutes of doc lookup saves 30 minutes of follow-up.

Step 3-4 — Concern & classification

✅ DO — Quality input

"Customer 1234567 redeemed 500 points but wallet shows 600 deducted. /v2/wallet/redeem returns 200 but customer status log shows DOUBLE_DEBIT. Reproduced 3 times today."

Includes: customer ID, exact endpoint, expected vs actual, frequency. AI uses every concrete identifier as a search key.

❌ DON'T — Vague input

"Points issue"

No identifiers, no symptoms, no API. AI can't ground anything; result is generic "check the configuration" advice.

Step 5-6 — Evidence & AI analysis

STEP 5 · OPTIONAL

More context

Offered when classification confidence is below ~70%. Three inputs:

  • 📁 File upload — logs, HARs
  • 📋 Paste raw text — error dumps, console output
  • 📞 Go on a call — opens a notepad for the call transcript
STEP 6 · TWO CLAUDE CALLS IN PARALLEL

Solution analysis + triage

Solution analysis — RCA hypothesis grounded in SDD content.
Triage — priority + category with confidence.

  • 80%+ Trust unless you have specific reason
  • 50-80% Review reasoning; override if you disagree
  • < 50% AI hedging — override based on judgement

Read the confidence number, not just the result

"P1 / Bug / 87% confidence with quoted SDD section X" tells a different story than "P1 / Bug / 32%, reasoning: not enough information". Same output, very different reliability.

Step 7-8 — Create & follow-up

STEP 7

  • Environment customfield_11800

On click:

Best practices for 100% efficiency

1

Always go through the chatbot, not Jira directly

Jira-direct ticket creation skips: AI triage, classification, auto-emails, SLA, KB lookup, auto-processing. Chatbot's overhead is ~2 extra clicks; payoff is correctness and follow-up automation.

2

Save brand credentials the first time

One person's 90-second setup saves the whole team's 90 seconds × N future tickets. MD5-hashed, never re-displayed.

3

Index SDDs as you find them

Saved per-brand, shared with the whole team. Stops 3 engineers re-finding the same doc 3 times. First ticket on a new brand = 5-minute investment for everyone.

4

Override AI triage when you know better

Logs your override + reason. Frequent overrides on the same kind of ticket signal a tuning gap; rare overrides confirm AI is calibrated.

5

Click "Yes, Resolved" on the verification link

Closes the Jira ticket automatically. Updates resolution metrics. Skipping leaves tickets sitting in "Resolved" forever and skews MTTR.

Frequently asked questions

Can two engineers work on the same ticket simultaneously?

Yes. Both see the same brand_documents, KB, SLA. Comments fire in real-time on Jira.

What if I close my browser mid-flow?

Conversation state is lost client-side, but anything saved server-side (creds, docs, ticket) persists. Restart — saved data auto-loads.

Does the chatbot read my Atlassian password?

No. SSO uses OAuth 2.0 — chatbot never sees your password. Brand till passwords are stored as MD5 hash, never displayed back.

Who shows as Reporter on Jira when I create a ticket?

You. Ticket creation uses your OAuth session, so Reporter reflects the actual IM engineer. Comments and reads use the service account.

Difference between chatbot and PSV-IM-Agent?

Chatbot = the UI you use directly. Agent = background service that auto-processes P0/P1 tickets via Jira webhook, posts deeper code-grounded findings.

Does the AI use my data for training?

No. Anthropic's Claude API doesn't use customer prompts for training. Concern text and SDD content stay within the request/response loop.

Reaching 100% efficiency

Three habits separate occasional users from power users.

✓

Always finish the verification flow

"Yes, Resolved" or "No, still open" — closes the loop both for the customer and for the eval layer. Skipping leaves data orphaned and skews resolution metrics.

📚

Index docs as you find them

Treat the first ticket for any new brand as a 5-minute investment that benefits every future ticket on that brand. Confluence URL paste in Step 2 takes 30 seconds.

🧭

Read the triage confidence and the reasoning, not just outputs

When you understand why the AI said "P1 / Bug / SDD section X", you'll know when to trust it and when to override. The "why" is always shown — it's right there.

Net effect

Engineers who do these three consistently: ticket-handling time drops from ~30-60 minutes to ~5-10 minutes per concern, with measurably better triage accuracy and SLA compliance.
